# GitLab MR 템플릿

> GitLab Merge Request 작성 가이드 및 템플릿.
> 커밋 컨벤션과 리뷰 가이드를 MR 레벨에서 통합 적용한다.

> 서비스 무관 공통 규칙은 [커밋 컨벤션](../common/commit-convention.md), [코드 리뷰 가이드](../common/review-guide.md)를 참조한다.

---

## CLAUDE.md 삽입용

아래 블록을 프로젝트 CLAUDE.md에 복사하여 사용한다.

```
## GitLab MR

MR 제목: <type>(<scope>): <설명> (커밋 컨벤션 동일)
MR 크기: 변경 파일 10개 이하, +400줄 이하 권장
MR 본문: 반드시 '## 변경 사항', '## 테스트' 섹션 포함
MR 라벨: type 라벨 필수 (feat, fix, refactor 등)
Draft MR: 작업 중이면 Draft로 생성, 리뷰 준비 완료 시 Draft 해제
리뷰어: 최소 1명 지정 필수
```

## 1. MR 템플릿

### 기본 템플릿

`.gitlab/merge_request_templates/Default.md`에 저장하여 사용한다.

```markdown
## 변경 사항
<!-- 이 MR이 "왜" 필요한지 설명합니다. 어떤 문제를 해결하는지, 어떤 기능을 추가하는지 기술합니다. -->


## 변경 유형
- [ ] 새 기능 (feat)
- [ ] 버그 수정 (fix)
- [ ] 리팩토링 (refactor)
- [ ] 문서 (docs)
- [ ] 테스트 (test)
- [ ] 기타: ___

## 주요 변경 내용
<!-- 핵심 변경사항을 bullet으로 나열합니다. 리뷰어가 어디를 중점 확인해야 하는지 안내합니다. -->
-
-

## 테스트
<!-- 어떻게 테스트했는지 기술합니다. -->
- [ ] 단위 테스트 추가/수정
- [ ] 통합 테스트 추가/수정
- [ ] 수동 테스트 완료

## 체크리스트
- [ ] 커밋 메시지가 컨벤션을 따름
- [ ] self-review 완료
- [ ] 불필요한 console.log / print / debugger 제거
- [ ] Breaking Change 여부 확인 및 고지
- [ ] 관련 문서 업데이트 (필요 시)

## 관련 이슈
<!-- Closes #이슈번호 -->

## 스크린샷 (UI 변경 시)
<!-- Before / After 캡처를 첨부합니다. -->
| Before | After |
|--------|-------|
|        |       |
```

### 각 섹션 작성 가이드

| 섹션 | 필수 | 작성 기준 |
|------|------|----------|
| 변경 사항 | 필수 | "왜" 중심. 이슈 링크만으로 대체하지 않는다. 최소 2-3문장. |
| 변경 유형 | 필수 | 하나만 선택. 여러 유형이면 MR 분할을 고려한다. |
| 주요 변경 내용 | 필수 | 리뷰어에게 "어디를 볼지" 안내. 파일명/함수명 포함. |
| 테스트 | 필수 | 테스트 방법과 결과. 자동 테스트가 없으면 수동 테스트 절차 기술. |
| 체크리스트 | 필수 | 모두 체크 후 리뷰 요청. 체크되지 않은 항목은 사유 설명. |
| 관련 이슈 | 권장 | `Closes #123` 키워드로 머지 시 자동 클로즈. |
| 스크린샷 | 조건부 | UI 변경이 있는 경우 필수. Before/After 비교. |

## 2. MR 제목 규칙

커밋 컨벤션과 동일한 형식을 사용한다:

```
<type>(<scope>): <설명>
```

| 예시 | 설명 |
|------|------|
| `feat(auth): OAuth2 로그인 추가` | 새 기능 |
| `fix(cart): 빈 장바구니 null 오류 수정` | 버그 수정 |
| `refactor(api): 검증 로직 분리` | 리팩토링 |
| `docs: API 사용 가이드 업데이트` | 문서 |

**규칙**:
- Squash Merge 시 MR 제목이 최종 커밋 메시지가 되므로, 컨벤션 준수가 중요하다.
- Draft MR은 제목 앞에 `Draft:` 접두사가 자동 추가된다 (수동 추가 불필요).

## 3. MR 크기 가이드

| 규모 | 변경 줄 수 | 파일 수 | 권장 |
|------|-----------|---------|------|
| S | ~100줄 | ~3개 | 가장 이상적 |
| M | ~300줄 | ~7개 | 적절 |
| L | ~500줄 | ~10개 | 분할 검토 |
| XL | 500줄+ | 10개+ | **분할 필수** |

### 분할 전략

큰 변경을 작은 MR로 나누는 방법:

| 전략 | 예시 |
|------|------|
| **레이어별** | DB 마이그레이션 → API → UI 순서로 3개 MR |
| **기능별** | 로그인 UI → 로그인 API → 세션 관리 순서로 3개 MR |
| **리팩토링 선행** | 리팩토링 MR 먼저 → 기능 추가 MR |
| **인터페이스 우선** | 인터페이스/타입 정의 MR → 구현 MR |

**원칙**: 각 MR은 독립적으로 머지 가능해야 한다. 중간 상태에서 빌드가 깨지면 안 된다.

## 4. 라벨 체계

### 필수 라벨

| 라벨 | 용도 | 값 |
|------|------|-----|
| `type::*` | 변경 유형 | `type::feat`, `type::fix`, `type::refactor`, `type::docs`, `type::chore` |
| `workflow::*` | 진행 상태 | `workflow::in-review`, `workflow::approved`, `workflow::blocked` |

### 선택 라벨

| 라벨 | 용도 | 값 |
|------|------|-----|
| `priority::*` | 우선순위 | `priority::critical`, `priority::high`, `priority::medium`, `priority::low` |
| `size::*` | MR 크기 | `size::S`, `size::M`, `size::L`, `size::XL` |
| `breaking` | Breaking Change 포함 | (단일 라벨) |

## 5. 리뷰어 지정 규칙

| 상황 | 리뷰어 수 | 기준 |
|------|----------|------|
| 일반 변경 | 1명 이상 | 해당 모듈 담당자 또는 최근 기여자 |
| 아키텍처 변경 | 2명 이상 | 모듈 담당자 + 테크 리드 |
| 보안 관련 변경 | 2명 이상 | 보안 담당자 포함 |
| 문서만 변경 | 1명 | 빠른 승인 |

**GitLab 기능 활용**:
- `CODEOWNERS` 파일로 모듈별 자동 리뷰어 지정
- Approval Rules로 필수 승인 조건 설정

## 6. FAQ

**Q: Draft MR은 언제 사용하나요?**
A: 작업 중 조기 피드백이 필요할 때, 또는 CI 파이프라인을 미리 확인하고 싶을 때. Draft 상태에서는 머지 버튼이 비활성화된다.

**Q: MR에 커밋이 너무 많으면?**
A: Squash Merge를 사용하면 MR의 모든 커밋이 하나로 합쳐진다. 이때 MR 제목이 최종 커밋 메시지가 되므로 제목 컨벤션이 중요하다.

**Q: 이슈 없이 MR을 만들어도 되나요?**
A: 소규모 변경(typo, 설정 변경 등)은 이슈 없이 가능하다. 기능 추가나 버그 수정은 이슈를 먼저 생성한다.
