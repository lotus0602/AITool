# 코드 리뷰 가이드

> 인간 리뷰어와 AI 리뷰어 모두를 위한 코드 리뷰 가이드.
> 일관된 리뷰 품질을 유지하고, 리뷰 코멘트의 명확한 소통을 보장한다.

---

## CLAUDE.md 삽입용

아래 블록을 프로젝트 CLAUDE.md에 복사하여 사용한다.

```
## 코드 리뷰

리뷰 코멘트 접두사:
  [must]     — 반드시 수정 (머지 차단)
  [should]   — 강력 권장 (사유 있으면 유지 가능)
  [nit]      — 사소한 개선 (선택)
  [question] — 이해를 위한 질문
  [praise]   — 좋은 코드 칭찬

리뷰 우선순위: 보안 > 버그 > 설계 > 성능 > 가독성 > 스타일
리뷰 범위: 변경된 코드 + 변경이 영향을 미치는 기존 코드

AI 리뷰 시:
- 관대하게 판단하지 않는다 (합격 편향 주의)
- [must]와 [should]를 엄격히 구분한다
- 코드에 있는 것뿐 아니라 "빠진 것"도 지적한다
- 컨벤션 위반은 [must]로 처리한다
```

## 1. 리뷰 원칙

### 리뷰의 목적

- **품질 보장**: 버그, 보안 취약점, 설계 결함을 배포 전에 발견한다.
- **지식 공유**: 코드 변경을 팀이 함께 이해한다.
- **일관성 유지**: 프로젝트 컨벤션과 패턴을 지킨다.

### 리뷰어의 책임

- 24시간 이내 첫 리뷰를 제공한다.
- 코드의 "의도"를 먼저 이해한 후 리뷰한다.
- 대안을 제시할 때는 구체적인 코드 예시를 포함한다.
- 승인할 때는 "무엇을 확인했는지" 명시한다.

### 작성자의 책임

- 머지 요청(MR/PR) 설명에 변경 동기와 테스트 내용을 기술한다.
- 리뷰 요청 전 자기 리뷰(self-review)를 수행한다.
- 리뷰 코멘트에 72시간 이내 응답한다.
- [must] 코멘트는 반드시 수정하고, [should]는 사유와 함께 유지 가능하다.

## 2. 리뷰 체크리스트

### 기능 (Functionality)

- [ ] 요구사항/이슈에 명시된 기능을 올바르게 구현했는가?
- [ ] 정상 경로(happy path)가 동작하는가?
- [ ] 에러 경로(error path)를 적절히 처리하는가?
- [ ] 경계값(edge case)을 고려했는가?
- [ ] 기존 기능에 의도치 않은 영향(regression)이 없는가?

### 설계 (Design)

- [ ] 변경의 범위가 적절한가? (너무 크면 분할 요청)
- [ ] 기존 패턴과 아키텍처를 따르는가?
- [ ] 불필요한 복잡도를 추가하지 않았는가?
- [ ] 적절한 추상화 수준인가? (과도 추상화 / 부족 모두 확인)
- [ ] 의존성 방향이 올바른가?

### 코드 품질 (Code Quality)

- [ ] 변수/함수/클래스 이름이 의도를 명확히 드러내는가?
- [ ] 중복 코드가 없는가? (기존 유틸리티 활용 가능 여부)
- [ ] 커밋 메시지가 [커밋 컨벤션](./commit-convention.md)을 따르는가?
- [ ] 불필요한 주석, console.log, TODO가 남아있지 않은가?

### 테스트 (Testing)

- [ ] 변경된 로직에 대한 테스트가 추가/수정되었는가?
- [ ] 테스트가 정상 경로와 에러 경로를 모두 다루는가?
- [ ] 테스트가 독립적이고 반복 실행 가능한가?
- [ ] 기존 테스트가 모두 통과하는가?

### 보안 (Security)

- [ ] 사용자 입력을 적절히 검증/이스케이프하는가?
- [ ] 인증/인가 로직이 올바른가?
- [ ] 민감 정보(비밀번호, 토큰, 키)가 코드에 하드코딩되지 않았는가?
- [ ] SQL Injection, XSS, CSRF 등 OWASP Top 10 취약점이 없는가?
- [ ] 로그에 민감 정보가 노출되지 않는가?

### 성능 (Performance)

- [ ] N+1 쿼리 문제가 없는가?
- [ ] 불필요한 데이터 로딩(over-fetching)이 없는가?
- [ ] 대용량 데이터 처리 시 메모리/시간 복잡도가 적절한가?
- [ ] 캐싱이 필요한 곳에 적용되었는가?

## 3. 리뷰 코멘트 컨벤션

### 접두사 시스템

| 접두사 | 의미 | 머지 영향 | 예시 |
|--------|------|----------|------|
| `[must]` | 반드시 수정 필요 | 머지 차단 | `[must] SQL injection 취약점. prepared statement를 사용해야 합니다.` |
| `[should]` | 강력 권장 | 사유 있으면 유지 가능 | `[should] 이 로직은 별도 함수로 추출하면 테스트가 쉬워집니다.` |
| `[nit]` | 사소한 개선 | 영향 없음 | `[nit] 변수명 data보다 userProfile이 의도를 잘 드러냅니다.` |
| `[question]` | 이해를 위한 질문 | 답변 필요 | `[question] 이 타임아웃 값 5초의 근거가 있나요?` |
| `[praise]` | 좋은 코드 칭찬 | 영향 없음 | `[praise] 에러 복구 로직이 깔끔합니다.` |

### 좋은 코멘트 vs 나쁜 코멘트

| 좋은 코멘트 | 나쁜 코멘트 |
|-------------|-------------|
| `[must] 이 쿼리에 인덱스가 없어 users 테이블 풀스캔이 발생합니다. email 컬럼에 인덱스를 추가하세요.` | `이거 느릴 것 같은데요` |
| `[should] retryCount를 환경변수로 분리하면 운영 시 재배포 없이 조정 가능합니다.` | `매직넘버 쓰지 마세요` |
| `[question] 이 캐시 TTL 30분은 데이터 특성상 적절한가요? 실시간성이 중요하다면 줄여야 할 수 있습니다.` | `왜 30분?` |

**좋은 코멘트의 특징**:
1. **접두사**로 중요도를 명시한다.
2. **문제**를 구체적으로 설명한다.
3. **대안** 또는 **방향**을 제시한다.

## 4. AI 리뷰어 가이드라인

> 이 섹션은 [하네스 엔지니어링 원칙서](../../harness-engineering-principles.md)의 원칙 4(격리와 병렬성)에서 다루는 Evaluator 캘리브레이션을 코드 리뷰에 적용한 것이다.

### AI 리뷰 시 주의사항

AI 리뷰어는 기본적으로 **관대한 편향**을 가진다:

> *"I watched it identify legitimate issues, then talk itself into deciding they weren't a big deal and approve the work anyway."* — Anthropic

이를 교정하기 위한 규칙:

1. **합격 편향 경계**: 문제를 발견했으면 심각도를 낮추지 않는다.
2. **누락 검사**: 코드에 있는 것뿐 아니라 "빠진 것"도 지적한다 (에러 처리, 테스트, 검증).
3. **접두사 엄격 적용**: [must]와 [should]를 명확히 구분하고, 애매하면 [must]로 올린다.
4. **독립성**: 작성자가 아닌 별도 에이전트가 리뷰한다 (자기 코드 자기 리뷰 금지).

### Evaluator 캘리브레이션

AI 리뷰어의 판단 기준을 정렬하기 위한 few-shot 예시:

**예시 1 — 불합격 (must)**

```python
def get_user(user_id):
    query = f"SELECT * FROM users WHERE id = {user_id}"
    return db.execute(query)
```

판정: `[must]` — SQL injection 취약점. user_id가 외부 입력이면 임의 SQL 실행 가능.
수정: `db.execute("SELECT * FROM users WHERE id = %s", (user_id,))`

**예시 2 — 불합격 (must)**

```javascript
async function processPayment(amount) {
  const result = await paymentApi.charge(amount);
  return result;
  // 에러 처리 없음, 결제 실패 시 사용자에게 아무 피드백 없음
}
```

판정: `[must]` — 결제 API 호출에 에러 처리가 없음. 네트워크 오류, 결제 실패 모두 무시됨.

**예시 3 — 합격 (should)**

```python
def calculate_discount(price, rate):
    return price * (1 - rate)
```

판정: `[should]` — rate가 0~1 범위인지 검증이 없지만, 호출부에서 이미 검증하고 있다면 합격. 다만 방어적으로 검증을 추가하면 더 안전함.

**예시 4 — 합격 (nit)**

```java
public List<User> getActiveUsers() {
    List<User> result = new ArrayList<>();
    for (User u : allUsers) {
        if (u.isActive()) result.add(u);
    }
    return result;
}
```

판정: `[nit]` — stream API로 간결하게 작성 가능하나, 동작에 문제 없음. 스타일 선호.

### 합격/불합격 기준 요약

| 판정 | 기준 |
|------|------|
| **불합격 [must]** | 보안 취약점, 데이터 손실 가능, 프로덕션 장애 유발, 컨벤션 심각 위반 |
| **조건부 합격 [should]** | 설계 개선 필요, 테스트 부족, 성능 우려, 가독성 저하 |
| **합격 [nit]** | 스타일 선호, 미미한 개선, 대안 제안 |
| **합격 [praise]** | 잘 작성된 코드에 대한 인정 |

## 5. 리뷰 프로세스

### 플로우

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ 머지 요청 │ ──→ │  리뷰    │ ──→ │  수정    │ ──→ │  승인    │
│  생성    │     │  (검토)  │     │ (반영)   │     │ (머지)   │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
     │                │                │                │
     ↓                ↓                ↓                ↓
  self-review      접두사 사용       must 전부        CI 통과
  + 설명 작성     + 체크리스트      수정 완료       + 승인 확보
                    기반 검토
```

### 타임라인 가이드

| 단계 | 기한 | 비고 |
|------|------|------|
| 첫 리뷰 | 24시간 이내 | 전체 리뷰가 아니어도 최초 코멘트 |
| 코멘트 응답 | 72시간 이내 | 수정 또는 사유 설명 |
| 재리뷰 | 24시간 이내 | 수정 반영 후 |
| 최종 승인 | 리뷰 완료 즉시 | 불필요한 지연 금지 |

### 리뷰 규모 가이드

| 규모 | 변경 줄 수 | 예상 리뷰 시간 | 권장 |
|------|-----------|---------------|------|
| S | ~100줄 | 15분 이내 | 가장 이상적 |
| M | ~300줄 | 30분 이내 | 적절 |
| L | ~500줄 | 1시간 이내 | 분할 검토 |
| XL | 500줄 초과 | — | 분할 필수 |

## 6. FAQ

**Q: [must] 코멘트에 동의하지 않으면?**
A: 사유를 코멘트로 설명한다. 리뷰어와 합의가 안 되면 팀 리드나 제3자에게 의견을 구한다. [must]를 무시하고 머지하지 않는다.

**Q: AI 리뷰와 인간 리뷰를 어떻게 조합하나요?**
A: AI가 1차 리뷰(컨벤션, 보안, 패턴 확인)를 수행하고, 인간이 2차 리뷰(설계 판단, 비즈니스 로직)를 수행하는 것이 효율적이다. AI 리뷰 결과를 인간 리뷰어가 참고하되, 최종 승인은 인간이 한다.

**Q: 리뷰에서 너무 많은 [nit]이 달리면?**
A: [nit]이 과도하면 리뷰 피로를 유발한다. 리뷰어는 핵심 이슈에 집중하고, 스타일 관련은 린터/포매터에 위임한다.

**Q: 작은 변경도 리뷰가 필요한가요?**
A: 원칙적으로 모든 변경은 리뷰를 거친다. 다만 typo 수정, 문서 업데이트 등 위험이 낮은 변경은 간소화된 리뷰(1명 승인, 빠른 확인)로 진행할 수 있다.
