# 티켓 워크플로

> 이슈 트래커 서비스에 무관한 티켓 생명주기와 상태 전이 규칙.
> 티켓이 생성부터 완료까지 어떤 과정을 거치는지, 각 단계에서 누가 무엇을 하는지 정의한다.

---

## CLAUDE.md 삽입용

아래 블록을 프로젝트 CLAUDE.md에 복사하여 사용한다.

```
## 티켓 워크플로

상태 전이: Open → In Progress → In Review → Done
  - Open: 생성됨, 아직 착수 안 함
  - In Progress: 담당자가 작업 중
  - In Review: 머지 요청(MR/PR) 생성, 리뷰 대기
  - Done: 머지 완료, 수용 기준 충족
  - Blocked: 외부 의존성으로 진행 불가 (어떤 상태에서든 전이 가능)

담당자 규칙:
  - 착수 시 본인을 담당자로 지정
  - 담당자 없는 티켓은 팀 트리아지에서 배정

티켓-브랜치-MR 연결:
  브랜치명에 티켓 ID 포함: feature/PROJ-123-add-login
  MR 설명에 Closes #123 기재 → 머지 시 자동 완료
```

## 1. 상태(Status) 정의

### 상태 목록

| 상태 | 의미 | 진입 조건 | 책임자 |
|------|------|----------|--------|
| `Open` | 생성됨, 착수 전 | 티켓 생성 시 | 생성자/PM |
| `In Progress` | 작업 진행 중 | 담당자가 착수 시 | 담당자 |
| `In Review` | 리뷰 대기 중 | MR/PR 생성 시 | 담당자 + 리뷰어 |
| `Done` | 완료 | 머지 + 수용 기준 충족 | 담당자 |
| `Blocked` | 진행 불가 | 외부 의존성 발생 시 | 담당자 |
| `Closed` | 작업 없이 종료 | 중복, 무효, 취소 시 | PM/생성자 |

### 상태 전이 다이어그램

```
┌──────────┐     ┌──────────────┐     ┌──────────┐     ┌──────────┐
│   Open   │ ──→ │ In Progress  │ ──→ │In Review │ ──→ │   Done   │
│          │     │              │     │          │     │          │
└──────────┘     └──────────────┘     └──────────┘     └──────────┘
     │                │    ↑               │
     │                ↓    │               │
     │           ┌──────────┐              │
     │           │ Blocked  │              │
     │           └──────────┘              │
     │                                     │
     ↓                                     ↓
┌──────────────────────────────────────────────────┐
│                    Closed                        │
│        (중복/무효/취소 — 어떤 상태에서든 전이 가능)    │
└──────────────────────────────────────────────────┘
```

### 전이 규칙

| 전이 | 조건 | 누가 |
|------|------|------|
| Open → In Progress | 담당자가 작업 착수 | 담당자 |
| In Progress → In Review | MR/PR 생성 | 담당자 |
| In Review → Done | 머지 완료 + 수용 기준 충족 | 리뷰어/담당자 |
| In Review → In Progress | 리뷰에서 수정 요청 | 담당자 |
| * → Blocked | 외부 의존성 발생 | 담당자 |
| Blocked → In Progress | 차단 요인 해소 | 담당자 |
| * → Closed | 중복, 무효, 취소 | PM/생성자 |

**규칙**:
- 역방향 전이(Done → In Progress)는 원칙적으로 금지. 완료 후 문제 발견 시 새 티켓을 생성한다.
- Blocked 전이 시 반드시 차단 사유와 의존 대상을 코멘트로 남긴다.

## 2. 각 상태에서의 책임

### Open

| 역할 | 책임 |
|------|------|
| 생성자 | 티켓 컨벤션에 맞게 작성 ([티켓 컨벤션](./ticket-convention.md) 참조) |
| PM/리드 | 우선순위 확인, 트리아지(분류 및 배정) |

### In Progress

| 역할 | 책임 |
|------|------|
| 담당자 | 본인을 Assignee로 지정 |
| 담당자 | 브랜치 생성 (티켓 ID 포함한 네이밍) |
| 담당자 | 진행 중 이슈/질문은 티켓에 코멘트 |
| 담당자 | 3일 이상 진행 없으면 상태 업데이트 또는 도움 요청 |

### In Review

| 역할 | 책임 |
|------|------|
| 담당자 | MR/PR 생성, 티켓과 연결 |
| 리뷰어 | 24시간 이내 첫 리뷰 ([코드 리뷰 가이드](../../git/common/review-guide.md) 참조) |
| 담당자 | 리뷰 코멘트 72시간 이내 대응 |

### Done

| 역할 | 책임 |
|------|------|
| 담당자 | 수용 기준 충족 최종 확인 |
| PM/QA | 필요 시 검증 후 확인 |

### Blocked

| 역할 | 책임 |
|------|------|
| 담당자 | 차단 사유 코멘트 (무엇이, 왜, 누구에게 의존) |
| 담당자 | 차단 대상 티켓을 링크 |
| 담당자/리드 | 에스컬레이션 판단 |

---

## 3. 담당자(Assignee) 규칙

| 규칙 | 설명 |
|------|------|
| 착수 시 자기 지정 | In Progress 전환과 동시에 본인을 Assignee로 지정 |
| 1티켓 1담당자 | 주 담당자는 1명. 협업이 필요하면 코멘트로 소통 |
| 미배정 티켓 | 주기적 트리아지(주 1회 이상)에서 배정 |
| 담당자 변경 | 코멘트로 사유를 남기고 변경. 무단 변경 금지 |
| 장기 미착수 | Open 상태 7일 초과 시 트리아지에서 재배정 또는 우선순위 조정 |

## 4. 티켓-브랜치-MR 연결

### 전체 흐름

```
┌──────────┐     ┌───────────────────────┐     ┌─────────────┐     ┌──────────┐
│  티켓    │ ──→ │       브랜치          │ ──→ │   MR/PR     │ ──→ │  머지    │
│ PROJ-123 │     │ feature/PROJ-123      │     │ Closes #123 │     │ 티켓     │
│ Open     │     │ -add-login            │     │             │     │ Done     │
└──────────┘     └───────────────────────┘     └─────────────┘     └──────────┘
     │                    │                          │                   │
     ↓                    ↓                          ↓                   ↓
  티켓 ID 확인      브랜치명에             MR 설명에            머지 시
                   티켓 ID 포함           티켓 연결 키워드      티켓 자동 완료
```

> 브랜치 네이밍 규칙은 [브랜치 전략](../../git/common/branch-strategy.md)을 참조한다.

### 연결 규칙

| 단계 | 규칙 | 예시 |
|------|------|------|
| 브랜치 생성 | 티켓 ID를 브랜치명에 포함 | `feature/PROJ-123-add-login` |
| 커밋 | footer에 티켓 참조 (선택) | `Refs #123` |
| MR/PR 생성 | 설명에 연결 키워드 포함 | `Closes #123` |
| 머지 | 티켓 자동 완료 (서비스 설정에 따라) | 티켓 → Done |

**원칙**: 모든 코드 변경은 티켓으로 추적 가능해야 한다. 티켓 없는 브랜치/MR은 원칙적으로 생성하지 않는다 (typo 등 사소한 변경 제외).

## 5. 에스컬레이션

### Blocked 상태 처리

| 경과 시간 | 대응 |
|----------|------|
| 즉시 | 차단 사유 코멘트 + 의존 대상 티켓 링크 |
| 24시간 | 차단 대상 담당자에게 직접 연락 |
| 48시간 | 팀 리드에게 에스컬레이션 |
| 72시간+ | 스프린트 범위 조정 논의 |

### 에스컬레이션 코멘트 형식

```
**Blocked**: [차단 사유]
**의존 대상**: #456 (담당자: @kim)
**예상 해소 시점**: 2026-04-10
**영향**: 이 티켓이 완료되지 않으면 릴리스 1.3.0 지연
```

## 6. 완료 기준 (Definition of Done)

티켓을 Done으로 전환하기 위한 공통 기준:

### 모든 유형 공통

- [ ] 수용 기준 / 완료 조건 충족
- [ ] 코드가 머지됨
- [ ] CI 파이프라인 통과
- [ ] 리뷰 승인 완료

### Bug 추가 기준

- [ ] 재현 절차대로 실행 시 문제가 해결됨
- [ ] 관련 테스트 추가 (같은 버그 재발 방지)

### Feature / Story 추가 기준

- [ ] 수용 기준 전체 통과
- [ ] 필요한 문서 업데이트 완료

### Spike 추가 기준

- [ ] 조사 결과 문서 작성 및 공유
- [ ] 팀 의사결정 완료

## 7. FAQ

**Q: In Review 상태에서 추가 작업이 필요하면?**
A: In Progress로 되돌린다. 리뷰에서 [must] 수정 요청을 받은 경우가 대표적. 수정 후 다시 In Review로 전환.

**Q: 하나의 티켓에 여러 MR을 만들어도 되나요?**
A: 가능하다. 큰 기능을 단계적으로 구현할 때 하나의 티켓에 여러 MR이 연결될 수 있다. 다만 티켓 분할을 먼저 검토한다.

**Q: Closed와 Done의 차이는?**
A: Done은 작업이 완료되어 종료. Closed는 작업 없이 종료 (중복 티켓, 더 이상 유효하지 않은 요구사항, 취소 등). Closed 시 사유를 코멘트로 남긴다.

**Q: 담당자가 휴가/부재 시?**
A: 진행 중 티켓은 트리아지에서 재배정한다. 담당자 변경 사유를 코멘트로 남긴다.

**Q: 티켓 없이 급하게 수정해야 할 때는?**
A: hotfix라도 티켓을 먼저 생성한다. 1분이면 만들 수 있다. 사후에 만들면 추적이 누락되기 쉽다.
