# 티켓 컨벤션

> 이슈 트래커 서비스(Jira, Linear, GitHub Issues 등)에 무관한 티켓 작성 규칙.
> 일관된 티켓 품질을 유지하고, AI와 인간 모두 티켓을 빠르게 이해할 수 있게 한다.

---

## CLAUDE.md 삽입용

아래 블록을 프로젝트 CLAUDE.md에 복사하여 사용한다.

```
## 티켓 컨벤션

제목 형식: [type] 구체적인 문제/기능 설명

type:
  bug     — 기존 기능의 오류/결함
  feature — 새로운 기능 추가
  task    — 기술적 작업 (리팩토링, 설정, 인프라 등)
  story   — 사용자 관점의 기능 요구사항
  spike   — 조사/검증/PoC (시간 제한 필수)

우선순위:
  critical — 서비스 장애, 데이터 손실 (즉시 착수)
  high     — 핵심 기능 장애, 다수 사용자 영향 (24시간 이내)
  medium   — 기능 불편, 우회 가능 (스프린트 내)
  low      — 개선사항, 미미한 영향 (백로그)

필수 항목: 제목, 유형, 우선순위, 설명, 수용 기준(Acceptance Criteria)
bug 필수: 재현 절차, 기대 동작, 실제 동작, 환경 정보
```

---

## 1. 티켓 유형

### type 목록

| type | 설명 | 사용 시점 | 커밋 type 대응 |
|------|------|----------|---------------|
| `bug` | 기존 기능의 오류, 결함, 의도와 다른 동작 | 프로덕션/스테이징에서 발견된 결함 | fix |
| `feature` | 새로운 기능 추가 | 기존에 없던 기능을 구현할 때 | feat |
| `task` | 기술적 작업 (리팩토링, 인프라, 설정 등) | 사용자에게 직접 보이지 않는 기술 작업 | chore, refactor, ci, build |
| `story` | 사용자 관점의 기능 요구사항 | 사용자 시나리오 기반 요구사항 정의 | feat |
| `spike` | 조사, 검증, PoC (Proof of Concept) | 기술 선택, 가능성 검증이 필요할 때 | docs |

> 커밋 type과의 대응은 [커밋 컨벤션](../../git/common/commit-convention.md)을 참조한다.

### feature vs story

| 구분 | feature | story |
|------|---------|-------|
| 관점 | 기술 중심 ("OAuth2 로그인 구현") | 사용자 중심 ("사용자가 소셜 로그인을 할 수 있다") |
| 크기 | 구현 단위 | feature보다 크거나 같음 |
| 분할 | 단독 작업 가능 | 여러 feature/task로 분할 가능 |
| 사용 | 개발팀 중심 프로젝트 | 기획/PM과 협업하는 프로젝트 |

**선택 기준**: 팀에서 story/feature를 구분할 필요가 없으면 `feature`만 사용해도 된다.

## 2. 제목 작성 규칙

### 형식

```
[type] 구체적인 문제/기능 설명
```

### 좋은 제목 vs 나쁜 제목

| 좋은 제목 | 나쁜 제목 | 이유 |
|-----------|-----------|------|
| `[bug] 로그인 시 비밀번호 5회 오류 후 계정 잠금 안 됨` | `로그인 버그` | 증상이 구체적이지 않음 |
| `[feature] OAuth2 Google 소셜 로그인 추가` | `소셜 로그인` | type 없음, 범위 불명확 |
| `[task] user 테이블 email 컬럼 인덱스 추가` | `DB 성능 개선` | 무엇을 하는지 알 수 없음 |
| `[spike] Redis vs Memcached 캐시 레이어 성능 비교` | `캐시 조사` | 조사 대상과 목적 불명확 |
| `[bug] 결제 완료 후 주문 상태가 '대기'로 유지됨` | `주문 상태 이상` | 어떤 상황에서 어떤 문제인지 불명확 |

### 제목 규칙

| 규칙 | 설명 |
|------|------|
| `[type]` 접두사 필수 | 목록에서 빠르게 유형 파악 |
| 50자 이내 권장 | 목록 뷰에서 잘리지 않도록 |
| 현상/기능을 구체적으로 | "어디서 무엇이 어떻게"를 담는다 |
| 솔루션이 아닌 문제를 기술 | "인덱스 추가" (X) → "사용자 조회 시 3초 이상 소요" (O) ... 단, task는 작업 자체를 기술해도 됨 |

## 3. 설명(Description) 작성 가이드

### Bug

```markdown
## 재현 절차
1. 로그인 페이지에서 잘못된 비밀번호를 5회 입력한다.
2. 6번째 로그인을 시도한다.

## 기대 동작
- 계정이 잠기고 "계정이 잠겼습니다" 메시지가 표시된다.

## 실제 동작
- 계정이 잠기지 않고 계속 로그인 시도가 가능하다.

## 환경 정보
- 브라우저: Chrome 120
- OS: macOS 14.2
- 서버: staging (v2.3.1)

## 스크린샷/로그
(해당 시 첨부)
```

**필수 항목**: 재현 절차, 기대 동작, 실제 동작, 환경 정보
**원칙**: 다른 사람이 이 티켓만 보고 재현할 수 있어야 한다.

### Feature

```markdown
## 배경
현재 이메일/비밀번호 로그인만 지원하여, 소셜 로그인을 원하는 사용자가 이탈하고 있다.

## 요구사항
- Google OAuth2 로그인을 지원한다.
- 기존 계정과 동일 이메일이면 자동 연결한다.
- 최초 소셜 로그인 시 프로필 보완 페이지로 이동한다.

## 수용 기준 (Acceptance Criteria)
- [ ] Google 로그인 버튼 클릭 시 OAuth2 인증 플로우 실행
- [ ] 인증 성공 시 JWT 토큰 발급
- [ ] 기존 이메일 계정과 자동 연결
- [ ] 최초 로그인 시 닉네임 입력 화면 노출

## 참고 자료
(디자인 시안, API 문서 링크 등)
```

**필수 항목**: 배경, 요구사항, 수용 기준
**원칙**: 수용 기준은 검증 가능한 체크리스트로 작성한다. "잘 동작한다"는 수용 기준이 아니다.

### Task

```markdown
## 목적
사용자 조회 API 응답 시간이 3초 이상 소요되어 성능 개선이 필요하다.

## 작업 내용
- user 테이블 email 컬럼에 B-tree 인덱스 추가
- 조회 쿼리에 불필요한 JOIN 제거

## 완료 조건
- [ ] 인덱스 추가 마이그레이션 작성 및 적용
- [ ] 조회 API 응답 시간 500ms 이내 확인
- [ ] 기존 테스트 통과
```

**필수 항목**: 목적, 작업 내용, 완료 조건

### Spike

```markdown
## 질문
캐시 레이어로 Redis와 Memcached 중 어떤 것이 적합한가?

## 조사 범위
- 읽기 성능 비교 (초당 10만 요청 기준)
- 데이터 구조 지원 (sorted set 필요 여부)
- 운영 복잡도 (클러스터링, 모니터링)

## 시간 제한
2일 (8시간 × 2)

## 산출물
- 비교 분석 문서 (Confluence/Wiki)
- 팀 공유 및 의사결정
```

**필수 항목**: 질문, 조사 범위, 시간 제한, 산출물
**원칙**: spike는 반드시 시간 제한(timebox)을 둔다. 끝없는 조사를 방지한다.

## 4. 우선순위

### 기준표

| 우선순위 | 영향 | 대응 시간 | 예시 |
|---------|------|----------|------|
| `critical` | 서비스 장애, 데이터 손실, 보안 사고 | 즉시 착수 | 결제 실패, 개인정보 유출, 서버 다운 |
| `high` | 핵심 기능 장애, 다수 사용자 영향 | 24시간 이내 | 로그인 불가, 검색 오류, 알림 미발송 |
| `medium` | 기능 불편, 우회 가능 | 현재 스프린트 내 | UI 깨짐, 느린 응답, 부정확한 메시지 |
| `low` | 개선사항, 미미한 영향 | 백로그 (여유 시 처리) | 텍스트 수정, 미미한 UI 개선, 코드 정리 |

### 영향범위 × 긴급도 매트릭스

```
              긴급도
              높음          낮음
영향  넓음    critical      high
범위  좁음    high          medium/low
```

| | 긴급 + 넓은 영향 | 긴급 + 좁은 영향 | 비긴급 + 넓은 영향 | 비긴급 + 좁은 영향 |
|---|---|---|---|---|
| 우선순위 | critical | high | high | medium/low |
| 예시 | 전체 서비스 다운 | 특정 사용자 결제 실패 | 검색 정확도 저하 | 관리자 페이지 UI 깨짐 |

## 5. 라벨/태그 체계

| 카테고리 | 용도 | 예시 |
|---------|------|------|
| `type` | 티켓 유형 | bug, feature, task, story, spike |
| `priority` | 우선순위 | critical, high, medium, low |
| `component` | 영향 받는 모듈/시스템 | auth, payment, search, infra |
| `platform` | 플랫폼 | web, ios, android, api |

**규칙**:
- `type`과 `priority`는 필수. 나머지는 프로젝트에 맞게 선택.
- 라벨은 소문자, 단일 단어 또는 케밥 케이스(kebab-case).
- 라벨 수를 최소한으로 유지한다. 아무도 안 쓰는 라벨은 제거한다.

## 6. 티켓 분할 기준

하나의 티켓이 너무 크면 추정이 어렵고, 리뷰가 늦어지고, 완료까지 오래 걸린다.

### 분할이 필요한 신호

| 신호 | 설명 |
|------|------|
| 작업 기간 3일 초과 | 한 스프린트 안에 완료되지 않을 가능성 |
| 설명이 스크롤 2페이지 이상 | 범위가 너무 넓음 |
| 수용 기준이 10개 초과 | 여러 기능이 섞여 있음 |
| "A하고 B도 하고 C도" | 접속사로 연결된 작업 = 별도 티켓 |
| 여러 모듈에 걸침 | 독립적으로 배포 가능한 단위로 분할 |

### 분할 전략

| 전략 | 예시 |
|------|------|
| **레이어별** | DB 변경 → API → UI 각각 별도 티켓 |
| **시나리오별** | 로그인 → 로그아웃 → 비밀번호 재설정 각각 별도 |
| **플랫폼별** | Web → iOS → Android 각각 별도 |
| **CRUD별** | 생성 → 조회 → 수정 → 삭제 각각 별도 |

**원칙**: 분할된 각 티켓은 독립적으로 완료 가능해야 한다. 의존성이 있으면 명시한다.

## 7. FAQ

**Q: story와 feature를 모두 써야 하나요?**
A: 아니다. 팀에서 사용자 스토리 기반으로 일하면 story를 쓰고, 기능 단위로 일하면 feature만 써도 된다. 둘 다 쓰는 경우 story → feature로 분할하는 관계가 일반적.

**Q: 우선순위를 누가 정하나요?**
A: 일반적으로 PM/PO가 비즈니스 영향을 기준으로 정하고, 개발팀이 기술적 긴급도를 반영한다. 합의가 안 되면 영향범위 × 긴급도 매트릭스를 기준으로 판단.

**Q: 수용 기준(Acceptance Criteria)은 반드시 있어야 하나요?**
A: feature와 story는 필수. 수용 기준 없이 구현하면 "완료"의 기준이 사람마다 달라진다. task는 완료 조건으로 대체 가능.

**Q: bug인지 feature인지 애매하면?**
A: "원래 이렇게 동작해야 하는데 안 된다"면 bug, "새로 이런 동작을 추가해달라"면 feature. 기존 스펙과 비교한다. 스펙이 없으면 팀에서 합의.

**Q: spike의 시간 제한을 초과하면?**
A: 시간 내에 결론을 내리지 못해도 그 시점까지의 조사 결과를 산출물로 정리한다. 추가 조사가 필요하면 새로운 spike 티켓을 생성한다. 시간 제한 없는 spike는 spike가 아니다.
